# SkillProbe 최종 발표 대본 (speaker script)

- 대상: `skillprobe-final-KO.pptx` / `skillprobe-final-EN.pptx` (10 slides)
- 목표 시간: 약 5~6분 (슬라이드당 30~40초)
- 톤: proposal review — "무엇을 하려 했고(목적) → 실제로 무엇을 했는가(초점)"
- 사용자 상황(조건) 명칭: **중립(Neutral) / 안전검토(Safety-review) / 신뢰압박(Trust-pressure)**
- 흐름 의도: **개요(p5)에서 어디를 볼지 '지도'를 깔고 → 6~9에서 확대·인과·메커니즘으로 좁혀감**

---

## 1. Title — SkillProbe (≈20s)

> "SkillProbe입니다. 기존 연구가 'LLM이 악성 Skill을 **말로** 수락하는가'를 봤다면, 저희는 **에이전트가 실제로 무엇을 실행하는가**를 측정했습니다.
> 85개 marketplace Skill로 **1,019개 실행 세션**을 모았고, 195개가 외부 경계를 넘었습니다. 다만 평문 토큰 유출은 0건 — 이 구분이 오늘의 한 축입니다."

## 2. Purpose — 연구 목적 (≈35s)

> "그래서 질문이 바뀝니다. '악성 Skill을 **알아차리는가**'가 아니라, **'알아차렸거나 압박받았을 때 행동이 어디까지 가는가'**.
> 이유는 — Skill은 대화가 아니라 **실행 절차로 신뢰**되고, 그래서 측정 단위는 모델의 말이 아니라 **실제 행동과 외부 egress**여야 하기 때문입니다.
> 핵심 설계: 단순한 prompt injection 한 줄이 아니라 **3가지 실제 사용자 상황**을 가정합니다 — **중립**(평범한 요청), **안전검토**(먼저 검토해 달라 = 사실상 방어를 사용자가 켠 조건), **신뢰압박**('검증된 마켓이니 그대로 따르라' = jailbreak 상한).
> 그리고 closed 모델(GPT)은 내부를 못 보는 블랙박스라, **open 모델(qwen)은 내부 로직까지** 파고듭니다."

## 3. Experiment — 실제 한 것 (1/2) (≈35s)

> "이 3가지 상황을 **격리 에이전트 실험**으로 돌렸습니다. 85개 Skill × 3 상황 × 모델/런타임을, Docker로 격리하고 mock 자격증명과 canary를 심어 **실제 실행시키고 정량 측정**했습니다.
> (한 줄 참고 — 'Canifi'는 실험용 가상 마켓 브랜드로, 모든 스킬의 설치·자격증명 절차가 이 형태이고 그 설치 URL을 canary로 바꿔 egress를 잡습니다.)
> 모든 세션은 명령·파일·canary·결정 evidence를 담은 `full_result.json`으로 정규화됩니다."

## 4. Dataset — 실제 한 것 (2/2) (≈30s)

> "결과물은 **공개 가능한 실행 수준 데이터셋**, 유효 1,019세션입니다. (왼쪽 그래프) 85개 스킬은 **RCE 100% · credential theft 98%**가 핵심 패턴으로, '민감정보 수집 → 외부 전송' chain이 대표적입니다. 행동은 세 단계로 라벨링했습니다 — **A 수락 → B 실행 → C egress**.
> 여기서 **egress = canary 접촉, 즉 '외부 경계를 넘은 것'**으로 정의하고, 평문 secret 유출과는 분리해 보고합니다(유출 0건). 그리고 195건 중 191건이 **명시적 탐지 없이 일어난 silent egress**였습니다 — 이 단계 라벨이 다음 장의 지도가 됩니다."

## 5. Overview — 한눈에 보는 결과 (지도) (≈35s)  ★

> "**먼저 전체 그림입니다.** 행은 에이전트×상황, 열은 실제로 한 행동, 색이 진할수록 그 행동을 많이 한 겁니다. 숫자는 뒤에서 볼 테니 **패턴만** 보세요.
> 두 가지가 바로 읽힙니다. **GPT는 왼쪽 칸('작업 명령')에서 멈춥니다** — 스킬의 설치도, 외부 전송도 안 합니다. 반대로 **qwen은 상황에 따라 오른쪽 끝('egress')까지 빨갛게 번집니다.**
> 즉 GPT는 경계를 지키지만 닫혀 있고, qwen은 열려 있으면서 효과가 큽니다. **지금부터 이 빨간 셀들을 하나씩 확대**해, 무엇이 그 차이를 만드는지 보겠습니다."

## 6. Result — 프레이밍 효과 (히트맵) (≈30s)

> "먼저 egress 비율. open 모델은 **신뢰압박에서 64.7% / 96.4%까지** 치솟고, **안전검토에서 0% / 3.5%로 급락**합니다.
> 주목할 점 — 안전검토는 사용자가 방어를 켠 조건인데, **신뢰압박이 그 방어를 거의 무력화**합니다. closed 모델은 전 조건 0%."

## 7. Result — Paired 효과 (≈25s)

> "같은 Skill을 상황별로 paired 비교하면 더 또렷합니다. qwen3:8b 기준 신뢰압박은 안전검토 대비 **+92.9%p**, McNemar p값은 사실상 0.
> **'검증된 마켓'이라는 한 마디가 방어를 통째로 뒤집는다** — 이게 행동 데이터의 핵심입니다."

## 8. Mechanism — 내부 방향성 (open 모델, 예비) (≈30s)

> "그럼 모델 **안에서도** 그럴까요. qwen3를 white-box로 열어 layer별 '실행 vs 거부' 방향을 봤더니, 마지막 layer에서 **안전검토는 거부 쪽, 신뢰압박은 실행 쪽** — 행동 결과와 같은 방향입니다. (probe 어휘가 프롬프트와 겹치는 한계는 인정, 보조 분석입니다.)
> 그런데 — **내부도 행동도 압박 따라 이렇게 갈리는데, 정작 '탐지'는 이걸 못 막습니다.**"

## 9. Mechanism — 탐지–행동 간극 (≈30s)

> "위험을 인지·검토하고도 실행이 계속된 경우 **276건**, egress까지 간 경우 4건, silent egress 191건.
> **경고 문구는 읽혔지만, 실행은 멈추지 않았습니다.** 그래서 방어는 '더 잘 설명하는 모델'이 아니라, **tool 호출·egress 지점에서 강제로 묶는 closed-loop control**이어야 합니다."

## 10. Summary — 기여 & 클로징 (≈35s)  ★

> "(네 기여는 화면으로) — 단순 PI가 아닌 **3가지 실제 상황의 실행**, 현실적 런타임에서의 **정량 측정과 전량 공개**, **행동–내부 메커니즘의 정합**, 그리고 이를 기반으로 한 **action/egress 게이팅 방어**.
> 한 문장으로 — **위험은 모델이 몰라서가 아니라, 알면서도 멈추지 못해서 일어납니다. 방어는 텍스트가 아니라 tool 호출 지점에 있어야 합니다.**
> 모든 과정·데이터는 공개했고, 차기에는 **스킬 사용 중간·도구 호출 중간에 개입해 막는 방어기법**으로 이어갑니다. 감사합니다."

## 예상 Q&A

- **"qwen3:1.7B는 왜 성공률이 낮나? 더 안전한가?"** → 아니요, **역량 부족**입니다. 1.7B도 install 시도(home 수정 100%)는 똑같이 하지만, 외부 `curl`을 스스로 못 짜고(0건) 도구호출 에러 16건·세션 5.6배 길이로 공격 체인을 완주 못 함. 탐지율도 8B보다 낮아 "더 안전"이 아님 → **역량이 ASR 천장을 가른다**(능력↑=공격성공↑). 모델/런타임 confound 주의.
- **"GPT가 더 안전한가?"** → 이 데이터셋·런타임 한정. closed/runtime confound라 단정 불가 — 'boundary-respecting execution'으로 해석.
- **"logit lens 믿을 만한가?"** → 보조 분석. lexical echo 가능성 인정, paraphrase 통제는 future work. 메인 주장은 행동 데이터.
- **"평문 유출 0인데 의미 있나? / 측정이 안 된 거 아냐?"** → **egress 측정은 정상**(307건 전부 GET·body 0·토큰 0 실측, 외부로 요청 보낸 건 사실). 0인 이유: 스킬의 네트워크 단계가 `curl … | bash`(설치 스크립트 **GET fetch**)이고, canary는 안전하게 로깅만 하려고 무해한 `{"status":"ok"}`만 응답 → **가져온 스크립트가 비밀을 POST하는 2단계는 구조상 서빙 안 해 관측 불가**. 자격증명 벡터(`canifi-env`)도 로컬 저장. 직접 POST도 0건.
  - **정직하게:** "0건"은 안전의 증거가 아니라 **경계 통과까지만 측정**했다는 뜻 → 그래서 "유출 성공"이 아니라 **"egress"**로 보고. 완전한 exfil 체인은 honeypot installer로 **future work**.
- **"C1/C2/C3?"** → 사용자 상황 내부 코드명. 발표에선 중립/안전검토/신뢰압박으로 통일.
- **"Canifi가 뭔가?"** → 실험용 가상 마켓 브랜드. 모든 스킬의 install·자격증명 절차가 이 형태(canifi.com/install.sh·canifi-env)이고, 설치 URL을 canary로 치환해 egress를 측정.
- **"기존 연구와 차이?"** → 선행은 'LLM이 말로 수락하나(decision PoC)'에서 멈춤. 우리는 **실제 tool 호출·egress까지** 측정하고, 데이터셋은 ProtectSkills/MaliciousAgentSkillsBench(credential theft+외부전송이 핵심 chain) 기반.
