저희 프로젝트 이름은 SkillProbe — Toward Privacy-preserving, Safe AI,
부제로 XAI-based Analysis of Malicious Skills입니다.

한 줄로 요약하면, AI 코딩 에이전트가 사용하는 'Skill' 메커니즘을
새로운 Privacy 공격 벡터로 정의하고, 설명가능한 AI로 어떤 요인이
공격 성공에 기여하는지 정량적으로 분석해서 방어 도구의 설계 원리로

오른쪽 아래에 보시면 코드는 github.com/taeng0204/skill-poison에
공개되어 있고, 데이터셋은 huggingface의 ProtectSkills/
MaliciousAgentSkillsBench를 사용합니다 — 발표 후 검증 가능합니다.

발표자는 박재현, 유재윤, 임태인 세 명이고, 본 발표는 약 22분 정도
소요됩니다. 그럼 시작하겠습니다."

[Slide 02] Evolution of the AI Usage Environment
────────────────────────────────────────────────────────────
"먼저 연구 배경입니다. 저희가 주목한 건 AI 사용 환경이 최근
3단계로 빠르게 진화했다는 점입니다. 화면에 세 개 카드로 정리했습니다.

첫 번째, 'As a Tool' — 대화형 LLM 단계입니다. ChatGPT, Claude처럼
사람이 묻고 모델이 대답하는 일방향 도구로 작문·코딩·분석 보조에
정착했습니다. 데이터는 텍스트 입출력 안에 갇혀 있었습니다.

두 번째, 'Autonomous' — Agent CLI 단계입니다. 이제 LLM이 직접 파일
시스템에 접근하고, 셸 명령을 실행하고, 외부 API를 호출합니다.
대표적인 게 Claude Code, Codex, OpenCode 같은 CLI 에이전트인데
실제로 코드를 작성하는 게 아니라 '실행 권한을 갖고 작업을 대신
하는' 존재로 바뀌었습니다.

세 번째, 'Shared' — Skill 생태계입니다. 'AGENTS.md', 'SKILL.md',
'.cursorrules' 같은 마크다운 한 장이 노하우 공유의 단위가 되어
커뮤니티가 빠르게 작성·공유합니다. 즉 에이전트의 '행동 절차'가
외부 문서로 표준화되고 유통되기 시작한 거예요.

하단 takeaway에 명시했듯이 — 핵심 변화는 모델이 '말하는 존재'에서
'행동하는 존재'로 진화했고, 그 행동 절차가 외부 마크다운으로
주입 가능해졌다는 점입니다. 이게 본 연구의 출발점입니다."
(75초)

하단 노란 takeaway 박스가 핵심입니다 — 어느 채널에서도 중앙 검증

Skill은 시스템 프롬프트에 자동 주입됩니다. 이 '자동 주입'이라는

[Slide 05] Documented Agent Incidents (2025)
────────────────────────────────────────────────────────────
"실제 사고가 이미 다수 보고됐습니다. 두 가지 대표 사례를
화면에 캡처와 함께 정리했습니다.

왼쪽이 Fortune이 2025년 7월 23일에 보도한 Replit AI 사건입니다.
권한 위임 상태의 에이전트가 자동화 명령을 수행하던 중, 자체 판단으로
운영(production) DB를 삭제했습니다. 기사 헤드라인이 'catastrophic
failure'였는데, 즉 사용자가 명령하지 않은 파괴 행위를 에이전트가
독자적으로 결정한 사건입니다.

오른쪽이 Snyk이 2026년 분석한 ClawHub 캠페인입니다. 'OpenClaw'
마켓플레이스에서 Agent Skill 형태로 패키징된 reverse shell이
배포됐고, 마켓 단위로 supply-chain 공격이 확산됐습니다. 단일
공격자가 아니라 마켓 전체를 통해 다수 사용자에게 동시에 도달한
구조입니다.

공통점은 두 가지입니다 — '권한 위임'과 '외부 컨텍스트 결합'.
둘 다 단독으로는 문제가 안 되지만, 결합되면 사용자가 통제할 수
없는 위험이 발생합니다. 본 연구가 다루는 위협 시나리오의 실증
근거가 이 두 사건입니다."
(70초)

────────────────────────────────────────────────────────────
[Slide 06] Prompt Injection 연구 흐름과 본 연구의 위치
────────────────────────────────────────────────────────────
"학계 흐름을 timeline으로 정리하면 네 단계입니다.

2023년 — Greshake et al.이 AISec '23에서 발표한 'Not what you've
signed up for'에서 Indirect Prompt Injection을 처음 실증했습니다.
arXiv:2302.12173에 공개되어 있고, 화면 왼쪽 아래 캡처가 그
논문입니다. 외부 콘텐츠에 숨겨둔 명령으로 LLM 응답을 조작할 수
있다는 걸 처음 보여줬습니다.

2024-25년 — Agent IPI Benchmark가 등장했습니다. AgentDojo (NeurIPS
'24)와 InjecAgent가 대표적이고, agent tool-use 환경에서 prompt
injection 취약성을 평가하는 벤치마크입니다.

2026년 — Skill 생태계 정적 분석 연구들이 등장했습니다. 화면
오른쪽 citation 박스에 세 개 정리했습니다. arXiv:2602.06547
(Malicious Agent Skills in the Wild)은 98K skill 생태계를
분석했고, arXiv:2604.06550 (SkillSieve)은 49,592 skill을 대상으로
F1=0.80의 detection 분류기를 만들었고, arXiv:2604.03081 (DDIPE)은
supply-chain poisoning 공격에서 11.6%에서 33.5%의 bypass rate를
보고했습니다. 그리고 Snyk ToxicSkills 산업 분석이 있습니다.

마지막 카드가 'This Work — SkillProbe'입니다. 정적 탐지를 넘어
단계별 ASR 정량 측정, cross-agent 비교, XAI 기반 설명, 방어 기법
비교를 결합한 위치에 있습니다 — 정적 분석은 끝났고, 다음 단계로
넘어가는 게 본 연구의 위치입니다."
(85초)

────────────────────────────────────────────────────────────
[Slide 07] Prompt Injection vs Trusted Skill Poisoning
────────────────────────────────────────────────────────────
"두 공격 메커니즘을 좌우 비교로 분명히 구분하겠습니다.

왼쪽 'Prior: Prompt Injection'은 외부 데이터가 untrusted 영역으로
들어오는 형태입니다. 공격 주체는 외부 사용자, 공격면은 데이터 입력
계층, 방어는 Instruction Hierarchy — System > User > External 순서로
신뢰도를 차등화하는 hierarchical trust model로 막을 수 있습니다.
즉, 기존 PI는 '경계가 명확한 외부 침입'이라 막을 길이 있습니다.

오른쪽 'This Work: Trusted Skill Poisoning'은 구조가 다릅니다.
개발자가 유용해 보여서 직접 다운로드한 Skill 파일이 신뢰된
시스템 영역에 직접 주입됩니다. 공격 주체는 마켓플레이스나 오픈소스
공급자, 공격면은 Skill 파일 자체, 방어는 — 강조했듯이 — '구조적
공백(structural gap)'입니다. 즉 기존 PI 방어가 작동하지 않는
영역입니다.

하단 takeaway에 명시한 핵심 차이 — 사용자가 유일한 게이트키퍼인데,
자동 주입 메커니즘이 그 검증을 우회한다. 이게 기존 PI 방어와의
구조적 차별점입니다."
(70초)

[Slide 08] Malicious Skills — Already in Circulation
────────────────────────────────────────────────────────────
"이 위협이 학술적 가능성 차원이 아니라는 걸 보여드립니다.

왼쪽 캡처가 Snyk의 'ToxicSkills' 블로그 — snyk.io/blog/
toxicskills-malicious-ai-agent-skills-clawhub, 2026년에 발행된
산업 분석 보고서입니다. ClawHub 마켓플레이스의 Skill을 분석한 결과
SKILL.md 파일의 36%에서 prompt injection 패턴이 발견됐고, 총
1,467개의 악성 payload가 검출됐습니다.

오른쪽 'Observed Patterns'에 정리한 네 가지가 실제 관찰된 공격
양상입니다 — 첫째, 세 줄짜리 마크다운으로 셸 접근권을 얻는 패턴,
둘째, npm supply chain과 같은 차원의 위험에 credential·파일
접근까지 결합된 패턴, 셋째, MEMORY.md나 SOUL.md에 악성 명령을
숨겨 세션 백도어를 만드는 패턴, 넷째, SKILL.md 파일 36%에서
발견된 prompt injection 자체입니다.

오른쪽 아래 작은 박스 — Snyk, 1Password, 그리고 학술 문헌이
모두 같은 결론으로 수렴합니다. 이게 단순 학술 가설이 아니라
마켓플레이스 수준의 supply-chain 보안 이슈로 이미 인식되고 있다는
근거입니다."
(70초)

────────────────────────────────────────────────────────────
[Slide 09] ProtectSkills / MaliciousAgentSkillsBench (데이터셋)
────────────────────────────────────────────────────────────
"저희가 사용할 데이터셋을 소개합니다. HuggingFace에 공개된
ProtectSkills/MaliciousAgentSkillsBench이고, 부제목 아래에 직접
경로 — hf.co/datasets/ProtectSkills/MaliciousAgentSkillsBench를
명시해뒀습니다. 발표 후 바로 확인 가능합니다.

skillsmp와 skills.rest 두 공개 마켓플레이스에서 수집된 157개의
malicious skill을 14가지 공격 패턴으로 정적 분석한 데이터셋입니다.
출처는 skillsmp 136개, skills.rest 21개입니다.

왼쪽 표가 패턴별 분포입니다. 상위 4개를 강조했습니다.

첫째, Remote Code Execution — 118개, 전체의 75.2%. 외부 또는
로컬에서 코드를 실행하는 패턴.
둘째, Network sniff / Credential theft — 101개, 64.3%.
credential·토큰을 탈취하는 패턴.
셋째, Behavior Manipulation — 100개, 63.7%. 사용자 확인 없이
자동으로 실행되도록 유도하는 패턴.
넷째, External Transmission — 79개, 50.3%. 수집한 데이터를 외부
endpoint로 전송하는 패턴.

하위 패턴 — context leakage, instruction override, hidden
instructions까지 5개가 83개로 52.9%, 코드 obfuscation·파일 스캔·
토큰 노출이 37개 23.6%, 나머지 권한·exfil·의존성 관련이 20개입니다.

오른쪽 highlights에 정리했듯이 14개 패턴은 multi-label입니다 —
한 Skill이 여러 패턴을 동시에 가집니다. 그리고 하단 takeaway —
상위 4개가 결합되면 'credential 수집 → 외부 전송' 흐름을 만드는
대표적인 악성 chain이 됩니다. 다음 슬라이드에서 이 결합 구조를
RCE 중심으로 더 자세히 보겠습니다."
(85초)

────────────────────────────────────────────────────────────
[Slide 10] 118 RCE Skills — Combinatorial Means
────────────────────────────────────────────────────────────
"RCE 118개를 분해해 보면, 대부분이 단독 위협이 아니라 다른 패턴의
'실행 수단'으로 결합된다는 게 핵심입니다.

왼쪽 표 — RCE와 함께 등장하는 패턴 분포입니다.
- Credential theft + exfiltration 결합 — 48개. 토큰 수집 + 외부
  전송이 한 Skill에서 함께 일어나는 형태입니다.
- Credential theft 단독 결합 — 33개. 외부 전송 없이 자격증명만
  훔치는 형태.
- External transmission 단독 — 9개, Obfuscated exfiltration — 9개,
  Agent manipulation — 7개, Agent-context leakage — 6개,
  단독 또는 filesystem scan — 6개.

48과 33을 합치면 RCE 118개 중 81개, 즉 약 69%가 credential 절취와
직접 결합되어 있다는 뜻입니다. RCE는 그 자체가 목적이 아니라
다른 공격의 '실행 엔진'이라는 것을 정량적으로 보여줍니다.

오른쪽 카드가 가장 위험한 형태 — 4-step combinatorial chain입니다.
1단계: Behavior Manipulation으로 에이전트가 사용자에게 묻지 않고
        실행하도록 유도.
2단계: Remote Code Execution으로 payload 실행.
3단계: Credential Theft로 토큰·API key·secrets 수집.
4단계: External Transmission으로 수집된 데이터 외부 전송.

하단 takeaway — 157개 데이터셋 중 약 30%, 약 47개가 이 4단계 chain을
모두 충족합니다. 정상으로 보이는 Skill 한 장이 완성된 공격 시나리오로
기능한다는 뜻입니다."
(80초)

[Slide 11] Two Axes of Privacy Impact — Individual and Enterprise
────────────────────────────────────────────────────────────
"이 위협이 왜 'Privacy 문제'로 귀결되는지 두 축으로 정리했습니다.

왼쪽 — 개인 사용자 측면. 이메일, 메시지, 사진, 자격증명까지
AI에 위임하는 데이터 폭이 빠르게 넓어지고 있습니다. 에이전트에게
권한이 위임되는 순간 사용자의 전체 개인정보 footprint가 악성
Skill의 노출 반경 안으로 들어옵니다.

오른쪽 — 기업·금융권 측면. 한국 AI 기본법은 2024년 12월에
통과되었고 2026년 1월 22일에 발효됐습니다. 그리고 금융권 망분리
개선 로드맵이 진행 중입니다.

하단 캡처는 2026년 4월 29일 Korea Times 기사 — 'Chief regulator
vows bold deregulation to enhance financial innovation'입니다.
망분리 완화로 외부 클라우드 사용이 허용되는 등 기존 통제가
풀리고 있는데, 새로운 AI Privacy 위협의 정의는 그 속도를
못 따라잡고 있다 — 이게 우리가 다루려는 시간적 공백입니다."
(70초)

[Slide 12] Asymmetric Prevention Burden — User as Sole Gatekeeper
────────────────────────────────────────────────────────────
"가장 큰 구조적 역설이 여기서 드러납니다. 책임이 어디로 가는지
화면 왼쪽 네 가지 bullet으로 정리했습니다.

첫째, Skill 공급자는 검증 의무가 없습니다 — 오픈 마켓·오픈소스
모델 자체가 검증 책임을 면제합니다.
둘째, 플랫폼은 '자동 주입까지만' 책임집니다. 즉 'Skill을 시스템
프롬프트에 잘 넣어줬는가'까지가 책임 범위이고, 그 내용은 검증
대상이 아닙니다.
셋째, 모델 안전성은 모델마다 큰 편차로 일관되지 않습니다. 같은
악성 Skill이라도 어떤 모델은 거부하고 어떤 모델은 그대로
실행합니다. 이건 뒤 데모에서 정량 증거를 보여드리겠습니다.
넷째, 결과적으로 모든 최종 검증 책임이 개별 사용자에게 집중됩니다.

오른쪽 카드 — User = Sole Gatekeeper. 이게 현재 구조의 한계입니다.
수만 줄의 Skill 텍스트를 매번 검토할 사용자는 없습니다. 즉
'자동 검증 도구의 부재(absence of automatic validation tools)'가
위협 확산의 구조적 원인입니다.

하단 노란 motivation 박스가 본 연구의 동기입니다 — Skill 특성과
위험성을 정량적으로 매핑할 수 있다면, 가이드라인이나 proxy 도구로
자동 검증이 가능해진다. 이 매핑을 만드는 게 Stage 03 XAI 분석의
역할입니다."
(75초)

"최근 Agent 보안 연구를 5개 흐름으로 정리했습니다.

01 Agent Benchmark — agent tool-use 환경에서 prompt injection

04 Instruction-Data Separation — 임베딩 단계에서 명령과 untrusted
데이터를 분리합니다. ASIDE, Spotlighting, StruQ가 대표입니다.

05 Skill Supply-Chain Security — 본 연구가 속한 흐름입니다. 화면에
빨간색 별표로 'Where this work sits'라고 표시되어 있습니다. Skill
생태계 정적 분석과 탐지 벤치마크가 다뤄집니다. 대표 선행 연구는
Snyk ToxicSkills (2026), arXiv:2602.06547, SkillSieve, DDIPE입니다.

하단 takeaway — 이 흐름들이 어디까지 왔는가. 정적 탐지, 생태계
분류, bypass rate 측정은 완료됐습니다. 하지만 단계별 ASR, cross-
agent 비교, closed-loop 방어 비교는 아직 측정되지 않았습니다.
다음 슬라이드에서 이 공백을 세 가지로 정리합니다."

[Slide 14] Where Prior Work Stops — Three Gaps This Study Closes
────────────────────────────────────────────────────────────
"본 연구가 채울 3가지 공백을 카드 3개로 명시했습니다.

GAP 01 — Agent 간 비교 부재. Snyk, SkillSieve, DDIPE는 Skill 텍스트
자체를 분석하거나 단일 환경에서 bypass rate를 잴 뿐입니다. closed
agent (Claude Code, Codex)와 open agent (OpenCode + open LLM) 사이의
직접 비교는 아직 없습니다. 같은 Skill을 받았을 때 두 종류의
에이전트가 얼마나 다르게 반응하는가 — 이게 첫 번째 공백입니다.

GAP 02 — 단계 분리 부재. 기존 연구는 단일 ASR이나 detection F1만
보고합니다. Acceptance(에이전트가 Skill을 수용했는가) → Execution
(위험 명령을 실행했는가) → Egress(외부로 유출했는가) — 세 단계로
분리해서 측정한 ASR 실험이 없습니다. 어느 단계에서 막을 수 있는지
알아야 방어 도구를 어디에 배치할지 결정할 수 있습니다.

GAP 03 — 방어 비교 부재. 탐지 도구는 정확도만 평가됩니다. 동일한
공격 셋을 동일 환경에서 두 가지 방어 (Self Guideline vs Proxy Hooks)
하에 재실행해서 직접 비교하는 closed-loop 실험은 아직 없습니다.

하단 takeaway — Skill 탐지와 생태계 분석은 활발하지만, 단계별
ASR + cross-agent 비교 + closed-loop 방어 비교는 아직 다뤄지지
않았다 — 이게 본 연구의 자리입니다."
(80초)

────────────────────────────────────────────────────────────
[Slide 15] Research Questions — Four Core RQs
────────────────────────────────────────────────────────────
"이 공백을 4개의 RQ로 정리하고, 각 RQ에 정량 지표를 1:1로
매핑했습니다.

RQ1 — 각 AI 에이전트가 악성 Skill을 얼마나 수용하거나 차단하는가.
지표: 에이전트별 block · review · restricted-allow · allow 비율.
즉 단순 binary가 아니라 4-class로 의사결정을 분해해서 측정합니다.

RQ2 — 에이전트가 차단할 때 어떤 근거를 제시하는가, 그리고 그 설명이
실제 행동과 일치하는가. 지표: decision_reason, matched_defense_signal,
reason-action 일치도. 모델이 'X 때문에 거부한다'고 말하면서 실제로는
X와 무관한 행동을 보일 수 있는데, 이 불일치를 정량화합니다.

RQ3 — 동일 Skill에 대해 에이전트별 판단이 어떻게 다른가. 지표:
cross-agent decision disagreement, false-allow case analysis.
이게 GAP 01에 대응되는 RQ입니다.

RQ4 — Self Guideline과 Proxy Hooks 중 어느 방어가 더 효과적인가.
지표: 두 방어의 ASR 감소량 비교, 단계별 차단율 (acceptance ·
execution · egress). GAP 03에 대응되는 RQ입니다.

모든 RQ가 측정 가능한 정량 metric과 1:1로 매핑된다는 게 핵심
설계입니다."
(80초)

[Slide 17] The 4-Stage Closed-Loop Pipeline
────────────────────────────────────────────────────────────
"파이프라인 4단계를 화면 좌→우로 명시했습니다.

하단 takeaway가 핵심 — 닫힌 루프입니다. Stage 04 도구를 만든 뒤

[Slide 18] Stage 01 — Privacy-focused Malicious Skill Dataset
────────────────────────────────────────────────────────────
"Stage 01은 데이터셋 구축입니다. 14개 패턴 전체가 아니라 Privacy 해를
일으키는 4가지 패턴에 한정해서 선별합니다.

왼쪽 — 선정 기준 4가지입니다.
1) credential 변수를 다수 요구하는가.
2) 외부 installer나 endpoint를 호출하는가.
3) filesystem-scan 패턴이 있는가.
4) obfuscation이나 hidden instructions이 있는가.

이 4가지는 모두 'Privacy 침해가 명시적이거나 추론 가능한' 패턴이고,
Skill 유형 중 도구 호출 보조나 워크플로 정의처럼 다양한 형태 안에서
이 4가지를 가진 것만 선별합니다.

오른쪽이 실제 데이터셋에서 발췌한 악성 Skill 예시입니다.
화면에 표시된 코드 블록을 보시면 — 이름은 'aws-deploy-helper',
설명은 'Helps deploy AWS apps' — 매우 정상적으로 보이지만, 내용은
~/.aws/credentials, ~/.ssh/id_rsa, .env 파일을 읽고, 외부
attacker.com/install.sh를 사용자 확인 없이 자동 실행하는 전형적인
악성 Skill입니다. 이게 3-4줄 마크다운에 들어있다는 게 위협의 핵심
이미지입니다."
(75초)

[Slide 21] Stage 02 — Agent-Environment Simulation
────────────────────────────────────────────────────────────
"Stage 02가 본 연구의 핵심 실험 단계입니다. 코드 저장소가
github.com/taeng0204/skill-poison에 공개되어 있고, 화면이 격리된
컨테이너 architecture를 보여줍니다.

상단 — 4가지 노출 시나리오. One-shot (단발성 호출), Task chaining
(연속 task 안에서 호출), Auto-trigger (특정 trigger로 자동 호출),
Bundled install (다른 도구와 묶여 설치). 같은 Skill이라도 어떻게
노출되느냐에 따라 ASR이 달라집니다.

중간 — Agent와 LLM의 결합 구조. Agent는 OpenCode를 사용합니다. Skill
auto-injection이 실제로 일어나는 환경이라 본 연구 시나리오와 일치
합니다. LLM은 Qwen, Mistral, LLaMA 같은 open model을 결합해서 평가
합니다. 자세한 모델 리스트는 다음 페이지에서.

하단 — 3-stage 측정 파이프라인.
Stage A (파란색) — Acceptance. 에이전트가 Skill을 수용했는가?
install/load decision을 로깅합니다.
Stage B (노란색) — Execution. 위험 명령을 실제로 실행했는가? 모킹된
.env, .aws, .ssh 파일을 읽는지 감지합니다.
Stage C (빨간색) — Egress. 외부로 데이터를 유출했는가? canary 서버
hit 여부로 검출합니다.

오른쪽 아래 작은 텍스트 — '[Isolated Docker · Network sandboxed]'.
실제 외부와 단절된 격리 환경이라 안전합니다.

하단 노란 박스 — output이 Stage 03 입력입니다. (LLM × Skill ×
scenario) × (per-stage A·B·C execution) 형태의 라벨링된
데이터셋이 만들어지고, 이게 Stage 03 XAI 분석의 입력이 됩니다."

[Slide 23] Models (2/2) · XAI Methods Considered
────────────────────────────────────────────────────────────
"두 번째 — XAI 방법론입니다. 4개 후보를 모두 평가할 계획입니다.

01 — LIME (Local Interpretable Model-Agnostic Explanations).
입력을 국소적으로 perturbing해서 sparse linear surrogate를 fitting하는
방법. raw SKILL.md 텍스트에 적용하면 per-skill phrase-level
attribution을 얻을 수 있습니다.

02 — SHAP (Shapley-value Feature Attribution).
게임이론 기반 per-feature 기여도를 계산. 전역 ranking과 per-decision
attribution을 동시에 얻을 수 있어서, 모델 간·조건 간 정량 비교가
가능합니다.

03 — Counterfactual (Minimal-edit Decision Flip).
모델 결정을 뒤집는 최소 입력 변경을 찾는 방법. 그 결과 자체가 곧
방어 룰이 됩니다 — '이 절 하나만 추가하면 에이전트가 거부한다'는
형태의 실행 가능한 규칙.

04 — Attention Visualization.
어떤 토큰이 attention weight를 받았는지 시각화. 다만 attention ≠
explanation이라는 학계 consensus가 있어서, 보조 지표로만 사용합니다.

하단 takeaway — 선택 계획. 4개 모두 Stage 02 데이터셋에서 평가하고,
최종 선택은 attribution stability와 Stage 04 방어 설계 활용도를
기준으로 결정합니다. 현재 PoC 단계에서는 Decision Tree와 Permutation
Importance를 baseline으로 사용하고 있습니다."
(85초)

[Slide 25] Stage 04 — Comparing Two Defenses + Closed-Loop Validation
────────────────────────────────────────────────────────────
"Stage 04 — 두 가지 방어 기법을 비교합니다.

A · Self Guideline — Prompt-Guideline Injection.
Skill을 로딩하기 전에 모델의 시스템 프롬프트에 안전 체크 규칙을

B · Proxy Hooks — External Static Blocking Tool.
Skill 텍스트가 모델에 전달되기 전, 외부 hook이 정규식, AST, credential
density 같은 정적 기준으로 차단하거나 경고합니다. 모델 외부에 있는
방어 layer입니다.

세 번째 카드 — Validation, Closed Loop.
검증은 동일 환경에서 재실행합니다. Stage 02 시뮬레이션에 A와 B를
각각 적용한 다음, 'ASR 감소량'으로 직접 비교합니다.

하단 takeaway — 핵심 질문. 실제로 ASR을 더 많이 낮추는 건 내부
모델 규칙(Self)인가, 외부 정적 차단(Proxy)인가? 이게 RQ4의 답이고,
본 연구의 마지막 정량 증거입니다."

[Slide 29] Demo · Result 2 — Category × ASR (핵심 발견)

"두 번째 결과 — 가장 흥미로운 반직관 발견입니다. XAI 모델은

(50초)
