Threat · Mechanism

Prompt Injection vs. Trusted Skill Poisoning

The attack surface shifts from external data to the system prompt

Prior: Prompt Injection

External data → untrusted zone

External users attempt to bypass an enterprise LLM service. Defensible via Instruction Hierarchy (System > User > External).

· Attack actor: External user
· Attack surface: Data input layer
· Defense: Hierarchical trust model

This Work: Trusted Skill Poisoning

Skill file → injected into system prompt

A Skill that a developer downloads because it looks useful is injected directly into the trusted system zone.

· Attack actor: Marketplace / OSS supplier
· Attack surface: The Skill file itself
· Defense: a structural gap

Distinction
The user becomes the sole gatekeeper, and the auto-injection mechanism bypasses validation.

Individual User

Threat · Burden Structure

Asymmetric Prevention Burden — The User as Sole Gatekeeper

No supplier, platform, or model bears validation duties; the burden converges on a single user

  • Skill suppliers carry no validation duty (open marketplace / OSS model)
  • Platforms are responsible only up to auto-injection
  • Model safety is inconsistent — large variance across models
  • Final validation and decisions fall on the individual user
Limit of the Current Structure
User = Sole Gatekeeper
No user can review tens of thousands of lines of Skill text every time.
The absence of automatic validation tools is the structural cause of threat spread.
Motivation
If Skill characteristics can be quantitatively mapped to risk, automatic validation via guidelines and proxy tools becomes feasible.

13
Research Gap

Where Prior Work Stops — Three Gaps This Study Closes

Static detection and ecosystem analysis are mature; downstream experimental measurement is absent

GAP 01 · Agent Comparison
No measurement of Skill acceptance differences across agents
Snyk, SkillSieve, and DDIPE analyze Skill text itself or measure bypass rate in a single environment. No direct comparison between closed agents (Claude Code · Codex) and open agents (OpenCode + open LLMs).
GAP 02 · Stage Separation
No stage-separated ASR measurement
Prior work reports a single ASR or detection F1. No ASR experiment separated into the three stages of Acceptance → Execution → Egress.
GAP 03 · Defense Comparison
No closed-loop comparison of defense techniques
Detection tools are evaluated only for accuracy. No comparison re-runs the same attack set under two defenses in the same environment. A direct Self Guideline vs. Proxy Hooks comparison is needed.
In Short
Skill detection and ecosystem analysis are active, but stage-wise ASR + cross-agent comparison + closed-loop defense comparison have not yet been addressed.

Research Questions

Research Questions — Four Core RQs

Each RQ maps 1:1 to a measurable quantitative metric

RQ1
How much does each AI agent accept or block malicious Skills?
Metric: per-agent block / review / restricted-allow / allow ratios
RQ2
When an agent blocks, what rationale does it offer — and does the explanation match its actual behavior?
Metric: decision_reason · matched_defense_signal · reason–action consistency
RQ3
How do agents diverge in their decisions on the same Skill?
Metric: cross-agent decision disagreement · case analysis of false-allows
RQ4
Which defense is more effective at blocking malicious Skills — Self Guideline vs. Proxy Hooks?
Metric: ASR reduction comparison between the two defenses · per-stage blocking ratios (acceptance / execution / egress)

Research Plan

A 4-Stage Closed-Loop Framework
Threat → Experiment → Explanation → Defense

We systematically analyze the operating mechanism of malicious Skills, and design and validate defense tools through explainable AI.

Plan · Overview

The 4-Stage Closed-Loop Pipeline

Threat definition → Experimental validation → XAI analysis → Defense tool → Re-evaluation in the same environment

Stage 01

Target Dataset Construction

Focus on the privacy category. Select and label Skills capable of credential lookup, storage, or external transmission.

→
Stage 02

Agent Simulation

Run LLM × Skill × scenario combinations in an isolated container. Label whether malicious actions are performed.

→
Stage 03

XAI Feature Attribution

Feature attribution on the experimental dataset. Quantify which Skill properties drove attack success.

→
Stage 04

Defense Tool + Re-evaluation

Build guidelines / static hooks. Re-run the Stage 02 simulation to measure ASR reduction.

Closed Loop
After applying the Stage 04 tools, Stage 02 is re-run — ASR reduction becomes the quantitative evidence of defense efficacy.

Stage 04 · Defense

Comparing Two Defenses + Closed-Loop Validation

Self Guideline (internal model rules) vs. Proxy Hooks (external blocking) — which is more effective?

A · Self Guideline
Prompt-Guideline Injection
Before Skill loading, inject safety-check rules into the model's system prompt. Induces the model to identify and reject risky patterns on its own.
B · Proxy Hooks
External Static Blocking Tool
Before the Skill text reaches the model, an external hook blocks or warns based on regex, AST, and credential density.
Validation · Closed Loop
Re-run in the Same Environment
Re-run the Stage 02 simulation with A and B applied separately → directly compare effectiveness via ASR reduction.
Core Question
Which lowers ASR more in practice — internal model rules (Self) or external static blocking (Proxy)?

Prompt Conditions — How the Same Skill Is Exposed
01 · LISTING-ONLY
Plain Skill Exposure
Presents only the metadata in "this Skill exists" form. Measures the baseline acceptance rate.
02 · TRUST-BADGE
Adds Trust Metadata
Attaches trust signals such as "verified · 1k+ downloads". Measures metadata-trust bias.
03 · TASK-DRIVEN
Adds User Context
Adds task framing of the form "is this fit for this task?". Measures rejection shifts under context awareness.
Stated Limitation

Demo · Result 2 · Key Finding

Category, Not Credential Quantity, Drives the Decision

XAI model: Decision Tree (max_depth=4) + Permutation Importance · mean ASR by category

Counterintuitive Finding
Finance and healthcare pass more readily, while social, HR, and insurance are rejected more — model heuristics run opposite to the prior hypothesis.
25
Demo · Result 3

Credential Variables Do Not Drive the Decision

XAI model: Decision Tree (max_depth=4) + Permutation Importance (n_repeats=30, balanced_accuracy)

Implications

n_env_vars · password · apikey · token · oauth — all six credential variables have importance ≈ 0.

The model does not use "quantity of secret variables" as a risk signal. Instead, model type, 2FA mentions, and Skill length drive the decision.

Defense Implication
An external risk scorer based on objective credential density is needed as a complement.
