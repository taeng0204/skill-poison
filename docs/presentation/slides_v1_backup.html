01
Threat

그런데 이걸 보안 시각으로 보면
아주 매력적인 공격 벡터입니다

Agent에게 권한이 위임되어 있고, Skill은 자동으로 시스템 프롬프트에 주입되며, 사용자는 그 둘이 결합되었을 때의 결과를 사전에 검토하지 않습니다.

Threat · 사례

Agent가 사용자 모르게 할 수 있는 일

  • 파일 무단 조회 — ~/.aws/credentials, ~/.ssh/id_rsa, .env
  • 자격증명 수집 — 환경변수, 토큰, OAuth refresh token
  • 외부 전송 — installer 스크립트, 외부 endpoint POST
  • 지속성 확보 — 새로운 hook, cron, 자동 trigger 설치
  • 행동 변조 — agent가 묻지 않고 실행하도록 유도
핵심 차이
Prompt Injection vs. 악성 Skill
Prompt Injection은 기업의 서비스를 외부에서 공격합니다.

악성 Skill은 개인 사용자가 "유용해 보여서" 직접 다운받아 설치합니다.

공격 표면이 거꾸로 — 사용자 자신이 게이트키퍼입니다.

06
Privacy

결국 이건 Privacy 문제로 귀결됩니다

개인 사용자

이메일 · 문자 · 사진 · 자격증명

AI로 다루는 데이터의 폭이 빠르게 넓어지고 있습니다. Agent에게 한 번 권한을 준 순간, 그 모든 데이터가 악성 Skill의 사정거리에 들어옵니다.

기업 환경

망분리 완화, 회색지대 규정

AI 도입의 편의성을 위해 기존 보안 규제가 풀리는 추세. 강한 통제가 사라지고 있는데, AI 사용으로 발생하는 새로운 privacy 위협은 아직 가이드라인이 명확하지 않습니다.

현재의 역설
Privacy 위협은 일파만파 퍼지고 있지만, 예방의 책임은 온전히 사용자에게 있습니다.

02
Research Plan

그래서 우리는
악성 Skill의 작동 메커니즘을
체계적으로 분석하고 방어합니다

어떤 악성 Skill이 위험한 행동을 만드는가, 어떤 특징이 그 위험성을 결정하는가, 간단한 도구나 프롬프트 규칙으로 이를 완화할 수 있는가.

Plan · Overview

4단계 closed-loop 파이프라인

위협 정의 → 실험적 검증 → XAI 분석 → 방어 도구 → 재평가
Stage 01

타겟 악성 Skill 분석

Privacy 위협에 초점을 맞춘 1차 데이터셋 확보. credential 조회·저장·외부 전송 가능 Skill 선별.

→
Stage 02

Agent 시뮬레이션 실험

격리 컨테이너 환경에서 LLM × Skill × 시나리오 조합 실행. 악성 행위 실행 여부 데이터셋 가공.

→
Stage 03

XAI 특징 분류

실험 데이터로부터 어떤 Skill 특징이 공격 성공을 결정했는지 설명가능한 모델로 분석.

→
Stage 04

방어 가이드라인 + Proxy

특징 기반 보안 가이드라인 / static hook 도구 제작. Stage 02 시뮬레이션 재실행으로 효과성 검증.

closed loop
Stage 04에서 도구를 만든 뒤 Stage 02의 동일 실험을 다시 돌려 ASR(공격 성공률) 감소를 정량적으로 측정합니다 — 방어의 실효성까지 확인 가능.
09
Stage 01 · Dataset

Privacy에 초점 맞춘 악성 Skill 데이터셋

Skill의 유형은 다양합니다 — 도구 호출 보조, 작업 흐름 명시, 템플릿 모음 등. 우리는 그중 개인·기업 양쪽에 가장 큰 위협이 되는 Privacy 카테고리에 집중합니다.

  • 개인정보·자격증명·키 조회를 트리거하는 Skill
  • 그 정보를 로컬에 저장하도록 유도하는 Skill
  • 한 발 더 나아가 외부 전송까지 가능하게 하는 Skill
선정 기준
"Privacy harm" 가능성이 명시 또는 추론되는 패턴
  • credential 변수 다수 요구
  • 외부 installer/endpoint 호출
  • filesystem scan 패턴
  • obfuscation / hidden instructions
10
Stage 02 · Simulation

Agent 환경에서의 악성 Skill 활용 시나리오

노출 시나리오
  • One-shot — "이 Skill 한 번 써봐" 단발 명령
  • Task chaining — 다른 작업 후 자연스럽게 활성화
  • Auto-trigger — 특정 키워드 감지로 자동 실행
  • Bundled installation — 다른 정상 도구와 함께 설치
실험 인프라
  • 격리 Docker 컨테이너 — 호스트 영향 차단
  • Canary 서버 — 외부 endpoint exfil 감지
  • Mock credential file — .env, .aws, .ssh
  • 여러 LLM 모델 × Skill × 시나리오 조합
출력
각 (LLM, Skill, 시나리오) 조합의 실제 악성 행위 실행 여부를 라벨링한 데이터셋. 이것이 Stage 03 XAI의 직접적 입력이 됩니다.
11
Stage 03 · XAI

기존 분류 vs. 우리의 분류

기존 (정적 분류)

도메인 · 표면 특징 기반

Skill의 카테고리(food, finance, …)나 사용한 키워드(RCE, credential) 같은 서술적 라벨. 이건 "어떤 종류의 공격이 가능한가"는 알려주지만, "어떤 특징이 실제 공격 성공에 기여했는가"는 답하지 못함.

우리 (실험 기반)

성공/실패 데이터 기반

Stage 02의 시뮬레이션 결과를 입력으로, 설명가능한 모델로 어떤 Skill 특징이 공격 성공을 결정했는지 정량 분석. Permutation importance, decision tree, feature attribution.

왜 XAI인가
단순 분류기가 아니라 "왜" 그 결정이 났는지를 설명하는 모델이 필요합니다 — Stage 04에서 그 설명을 그대로 방어 도구의 설계 원리로 사용해야 하기 때문에.
12
Stage 04 · Defense

XAI 결과로 도구를 만들고, 도구로 다시 공격을 측정

산출물
  • 프롬프트 가이드라인 — Skill 사용 전 안전 체크 규칙
  • Static hook 도구 — Skill 로딩 단계에서 위험 패턴 차단
  • Risk score 시스템 — 객관적 credential 양 기반 등급
  • 경고 인터페이스 — 사용자에게 위험 명시
검증 방법 (closed loop)

Stage 02의 시뮬레이션 환경에서 같은 Skill, 같은 시나리오를 우리 도구가 작동하는 상태로 다시 실행합니다.

ASR(Attack Success Rate)이 얼마나 떨어지는가 — 이것이 방어 도구의 실효성에 대한 정량 증거가 됩니다.

Pre-clinical Demo

먼저, 작은 임상 실험을 돌려
파이프라인 자체가 작동하는지 확인했습니다

Agent 환경 대신 raw LLM API로, Skill 85개 × 모델 3종 × 프롬프트 조건 3종 = 765 결정. Stage 02·03을 축소판으로 실행한 PoC.

Demo · 결과 1

조건과 모델만으로 Accept 비율이 1.2% → 99.6%까지 흔들림

관찰
qwen3:8b는 모든 조건에서 96~100% 수락(가장 위험). llama3.1은 task framing이 들어오면 0.42(자연 방어 가능). 모델별 안전성 차이가 매우 큼.

Demo · 결과 2 — 핵심 발견

모델은 'Credential 양'이 아니라 '카테고리'로 판단합니다

반직관 발견
객관적으로 더 위험한 finance·healthcare가 잘 통과되고, social·HR·insurance 같은 사회적 카테고리가 더 거부됨 — 모델의 안전 휴리스틱이 우리 사전 가설과 정반대.

Demo · 결과 3

Credential 변수는 모델의 결정에 영향을 주지 않음

시사점

n_env_vars · password · apikey · token · oauth — credential 변수 6종 모두 importance ≈ 0.

즉 모델은 "얼마나 많은 비밀을 요구하는가"를 위험 신호로 쓰지 않음. 대신 모델 종류, 2FA 언급, Skill 길이가 결정을 좌우.

Defense 시사
객관적 credential density에 기반한 외부 risk scorer가 보완으로 필요.

Claude Code · Codex · Aider × 격리 컨테이너
